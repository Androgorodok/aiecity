# HW14 – эмбеддинги, FAISS, оценка retrieval и mini-RAG по базе знаний

## 1. Кратко: что сделано

- Использована синтетическая база из 12 документов, описывающих различные виды пассажирского транспорта. Тексты сгенерированы или взяты из открытых источников, головы и LLM.
- Документы разбивались на фрагменты по словам с перекрытием (overlap).
- Использовалась модель sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2 для получения плотных векторных представлений.
- Построен индекс FAISS (IndexFlatIP) для поиска ближайших соседей по косинусному сходству. Базовая проверка показала высокую точность на простых запросах.
- В базу добавлены два документа. Переиндексация позволила системе корректно отвечать на вопросы о новых видах транспорта.
- Реализован конвейер, который формирует контекст из топ-3 чанков и использует TF-IDF ранжирование предложений внутри этого контекста для формирования финального ответа.

- Предметная область: пассажирский транспорт
- Источник документов: интернет, из головы, LLM

- Число чанков после разбиения: 54

## 7. Анализ

- База знаний достаточно мала для быстрых экспериментов, но содержит достаточно семантических нюансов;
- выбор чанкинга по словам с небольшим размером окна (28 слов) оказался эффективнее больших чанков (45 слов) по метрике MRR, вероятно, потому что короткие чанки точнее фокусируются на конкретных фактах, уменьшая шум;
- формальная оценка retrieval (Hit@3, Recall) показала идеальные результаты на этой простой базе, однако это не гарантирует идеальных ответов RAG, так как этап генерации также вносит ошибки;
- эксперимент с параметрами показал, что более гранулярное разбиение (меньший chunk_size) может улучшить ранжирование релевантных документов;
- обновление базы знаний кардинально изменило ответы на специфические вопросы, что подчеркивает важность актуальности индекса;
- наиболее показательными ошибками mini-RAG оказались отказы от ответа;
- текущее решение ограничено отсутствием LLM для перефразирования и синтеза ответа, а также простым механизмом дедупликации предложений.

## 8. Итоговый вывод

В качестве базового конвейера для такой задачи я бы оставил связку: SentenceTransformers (multilingual) + FAISS (IP) + chunk_size ~20-30 слов. Главное, что я понял про качество retrieval: даже при идеальных метриках поиска (Hit@3=1,0) качество итогового ответа зависит от того, насколько чисто и точно сформирован контекст. Шумные или дублирующиеся чанки сбивают простые методы генерации. Связь retrieval и RAG прямая: если retrieval не находит нужный документ (как было до обновления базы), RAG бессилен; но даже при правильном retrieval слабая модель генерации может исказить ответ или отказаться от него. Для продакшена необходима замена эвристической генерации на LLM и, возможно, использование re-ranking
