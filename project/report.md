Пользователями сервиса являются агрономы, фермеры и сельскохозяйственные предприятия, которые загружают фотографию листа через API и получают предсказание: здорово растение или поражено одной из трёх распространённых болезней: гельминтоспориоз, обыкновенная ржавчина, серая листовая пятнистость.

## 2. Постановка задачи и контекст

1. **Предметная область и задача:**
   - решением является классификация болезней листьев кукурузы по фотографиям, задача мотивирована тем, что визуальная диагностика болезней растений в поле требует экспертизы агронома, которая не всегда доступна оперативно;
   - потенциальными пользователями сервиса являются фермеры, которые загружают фотографии листьев кукурузы и получают вероятность заболевания той или иной болезнью.

2. **Формулировка задачи в терминах ML/ИИ:**
   - Входом является RGB-фотография листа кукурузы (после препроцессинга приводится к 224×224 пикселей, нормализуется по статистике ImageNet);
   - выходом является один из 4 классов — `Blight` (фитофтороз), `Common_Rust` (обычная ржавчина), `Gray_Leaf_Spot` (серая пятнистость), `Healthy` (здоровый лист) — вместе с вектором вероятностей (softmax) по всем четырём классам;
   - ограничениями являются точность (приоритетная характеристика, так как ошибочный диагноз болезни напрямую влияет на решения фермера по обработке посевов), скорость инференса (сервис должен отвечать за разумное время на одном изображении без GPU в продакшене, поэтому при выборе архитектуры баланс точность/скорость учитывался явно),  масштабируемость (сервис рассчитан на единичные запросы, массовая пакетная обработка не оптимизировалась отдельно).

3. **Целевые метрики качества:**
   - **Accuracy (test)** — базовая метрика для понимания общей корректности модели;
   - **F1 macro** — основная метрика для итогового сравнения моделей и экспериментов. Выбрана вместо accuracy, потому что классы датасета не сбалансированы, а macro-усреднение не позволяет высокому качеству на крупных классах маскировать слабый результат на редком классе;
   - **F1 по отдельным классам (особенно Gray_Leaf_Spot)** — выделена отдельно, так как именно этот класс оказался наиболее проблемным во всех проведённых экспериментах (наименьшее количество примеров в датасете и визуальное сходство с другими патологиями);
   - ROC-AUC и RMSE не использовались — первая менее интерпретируема при 4 классах без явной бинаризации, вторая не подходит для задачи классификации.

## 3. Данные

1. **Источник данных:**
   - использован открытый датасет с Kaggle: **Corn or Maize Leaf Disease Dataset** (автор — Smaranjit Ghose), доступен по ссылке: `https://www.kaggle.com/datasets/smaranjitghose/corn-or-maize-leaf-disease-dataset`;
   - датасет составлен из изображений листьев кукурузы, размеченных по 4 категориям и собран из открытых агрономических источников (включая PlantVillage и аналогичные базы) с экспертной проверкой соответствия класса заболеванию;
   - в датасете обезличенные данные и нет секретов.

2. **Структура данных:**
   - данные распределены по папкам, разбияния данных осуществляется в коде;
   - ключевыми признаками являются сами изображения листа, а целевой переменной — метка класса;
   - данные являются изображениями.

3. **Предобработка и EDA:**
   - данные были приведены к единому размеру, нормализировались и аугментированы, основные шаги EDA лежат в `notebooks/01_eda.ipynb`;
   - ключевое наблюдение EDA — дисбаланс классов, где распределение исходного датасета по классам не равномерно: `Gray_Leaf_Spot` — наименьший класс (574 изображения), тогда как остальные классы содержат заметно больше примеров, поэтому рассчитаны веса классов для компенсации дисбаланса в функции потерь.

## 4. Модели и подходы

1. **Базовые (baseline) модели:**
   - были использованы простые модели использованы в качестве baseline: Simple CNN, Resnet18, MobileNetV2, EfficientNet-B0 в `notebooks/02_baselines.ipynb`;
   - ResNet18 выбран как победитель baseline и базовая модель для дальнейших экспериментов.

2. **Улучшенные модели и эксперименты:**
   - на базе ResNet18 проведено 7 экспериментов, сгруппированных по трём гипотезам: борьба с переобучением (усиленные аугментации, Label Smoothing, Weight Decay), работа с дисбалансом класса Gray_Leaf_Spot (Weighted Loss, Oversampling), стратегии fine-tune (двухфазный fine-tune, Discriminative Learning Rate);
   - важные гиперпараметры: Label Smoothing (0,1), Weight Decay (1e-4), learning rate (1e-4 для backbone, 1e-3 для головы), аугментации (RandomRotation, HorizontalFlip, VerticalFlip, ColorJitter, RandomAffine, GaussianBlur, RandomErasing);
   - эксперименты показали, что ни одна из применённых техник не дала статистически значимого прироста над baseline (лучший результат — Label Smoothing с Test Acc 0.9785, что практически идентично baseline), двухфазный fine-tune оказался худшим (0.9523), что интерпретировано как недостаточное количество эпох для инициализации головы

3. **Нейросетевые модели (если применимо):**
   - использованные архитектуры: ResNet18, MobileNetV2, EfficientNet-B0 (предобученные на ImageNet) и простая свёрточная сеть Simple CNN (обучена с нуля);
   - особенности обучения: batch size 64 (в коде обучения по умолчанию — 32, расхождение отмечено в разделе ограничений), оптимизатор Adam, learning rate 1e-4 для fine-tune предобученных моделей и 1e-3 для Simple CNN, scheduler CosineAnnealingLR, размер изображения 224×224, аугментации, Early Stopping, фиксация random seed 42.

## 5. Экспериментальный протокол и результаты

1. **Экспериментальный протокол:**
   - Данные разделены на train/val/test с разбиением 80/10/10 соответственно с использованием стратифицированного `train_test_split` без k-fold;
   - Все метрики (accuracy, F1 macro, F1 по отдельным классам) считаются на test-выборке, не участвовавшей ни в обучении, ни в подборе гиперпараметров. Val-выборка используется только для выбора лучшей эпохи (checkpoint с максимальным val accuracy сохраняется и используется для финальной оценки на test) и для early stopping в экспериментах `notebooks/03_models_tuning.ipynb`.

2. **Сравнение моделей по метрикам:**

   Сравнивались 4 архитектуры в одинаковых условиях обучения в `notebooks/02_baselines.ipynb`:

| Модель / конфигурация | Описание | Test Accuracy | F1 macro | Комментарий |
|---|---|---|---|---|
| Simple CNN | Обучение с нуля, lr=1e-3 | 0.9451 | 0.9283 | Нижняя планка (lower bound) |
| MobileNetV2 | Full fine-tune, ImageNet pretrained, lr=1e-4 | 0.9642 | 0.9514 | Наименьшая модель (~3,5M параметров) |
| EfficientNet-B0 | Full fine-tune, ImageNet pretrained, lr=1e-4 | 0.9666 | 0.9547 | Среднее число параметров (~5,3M) |
| **ResNet18** | **Full fine-tune, ImageNet pretrained, lr=1e-4** | **0.9690** | **0.9591** | **Выбрана как победитель baseline** |

На базе ResNet18 проверено 7 конфигураций в `notebooks/03_models_tuning.ipynb`:

| Модель / конфигурация | Описание | Test Accuracy | F1 macro | F1 (Gray_Leaf_Spot) | Комментарий |
|---|---|---|---|---|---|
| Baseline ResNet18 | Полный fine-tune, без доп. техник | 0.9785 | 0.9717 | 0.9300 | Точка отсчёта |
| **Label Smoothing** | `label_smoothing=0.1` | **0.9785** | **0.9715** | **0.9273** | **Финальная модель** (формально лучший эксперимент) |
| Weighted Loss | Веса по инверсии частоты класса | 0.9737 | 0.9631 | 0.8991 | Чуть хуже baseline |
| Strong Augmentations | + GaussianBlur, RandomErasing | 0.9666 | 0.9541 | 0.8785 | Усложнение задачи не успело компенсироваться |
| Discriminative LR | backbone lr=1e-5, голова lr=1e-4 | 0.9618 | 0.9524 | 0.8929 | Слишком медленная сходимость backbone |
| Weight Decay | `weight_decay=1e-4` | 0.9642 | 0.9509 | 0.8704 | Early stopping сработал рано (эпоха 10) |
| Oversampling | `WeightedRandomSampler` | 0.9594 | 0.9498 | 0.8870 | Recall по GLS вырос, precision упал |
| Two-phase Fine-tune | 5 эпох голова → разморозка, lr=1e-5 | 0.9523 | 0.9377 | 0.8468 | Худший результат — недостаточно эпох на фазу 1 |

3. **Выбор финальной модели:**
   - в качестве финальной модели для развертывания в сервисе был выбран ResNet18 с Label Smoothing (0,1), потому что у неё выше F1 macro и Test Accuracy, техника способствует улучшению калибровки вероятностей модели, что важно для интерпретируемости предсказаний в пользовательском сценарии, в `notebooks/04_inference.ipynb` была протестирована данная модель;
   - ResNet18 стабильно показывал наивысшую точность, но при этом самый тяжёлый из трёх предобученных вариантов (~11M параметров против ~3,5–5,3M у альтернатив), что увеличивает время CPU-инференса в сервисе. Решение в пользу точности было принято осознанно, так как для задачи диагностики болезни ошибка модели важнее небольшого выигрыша в скорости ответа API.

## 6. Архитектура решения и сервис

1. **Архитектура пайплайна:**
   - загрузка данных из `splits/*.csv` и папок с изображениями формирует DataLoader, который применяет предобработку (ресайз, нормализацию, аугментацию) и подаёт батчи в модель (PyTorch, ResNet18). Обученная модель сохраняется в `artifacts/`, откуда её загружает сервис (FastAPI) для инференса через эндпоинт `/predict`.;
   - данные хранятся в папке `data/data/`, организованной по классам, а разбиение на train/val/test сохраняется в CSV-файлах в директории splits/. При загрузке через DataLoader применяются трансформации: ресайз до 224×224 пикселей, нормализация по статистикам ImageNet, а для обучающей выборки используются аугментации (повороты, отражения, изменение яркости и контрастности). В ходе экспериментов сравнивались четыре архитектуры: простая свёрточная сеть Simple CNN (обучение с нуля), а также предобученные на ImageNet ResNet18, MobileNetV2 и EfficientNet-B0. Лучшей архитектурой признан ResNet18, который стал основой для дальнейшего тюнинга. Проведено 7 экспериментов с различными техниками регуляризации и борьбы с дисбалансом классов: Label Smoothing, Weighted Loss, усиленные аугментации, Discriminative Learning Rate, Weight Decay, Oversampling и двухфазный fine-tune. Финальной моделью выбран ResNet18 с Label Smoothing, показавший Test Accuracy 97.85% и F1 macro 0.9715. Артефакты обучения (веса модели `best_model.pt` и конфигурация с классами config.json) сохраняются в папку `artifacts/`. Модель интегрирована в веб-сервис на FastAPI, который загружает сохранённые артефакты и предоставляет два эндпоинта: GET /health для проверки статуса и POST /predict для загрузки изображения и получения предсказаний с вероятностями по всем четырём классам. Интерактивная документация API доступна через Swagger UI. Наблюдаемость обеспечивается автоматическим логированием запросов и ошибок через Uvicorn, а также эндпоинтом healthcheck для мониторинга доступности сервиса. Пользователи могут не только использовать готовый сервис, но и самостоятельно запустить обучение модели на своих данных через `python -m src.train` или выполнить инференс на одном изображении через скрипт `src/inference.py`. Все параметры обучения (количество эпох, размер батча, скорость обучения, аугментации) легко изменяются в `src/train.py`, что позволяет адаптировать систему под новые классы болезней или региональные особенности без изменения основной логики проекта.


2. **API и endpoints:**
   - реализованы endpoints `/health` и `/predict`;
   - вход — multipart/form-data с полем `file` (изображение листа кукурузы, любой формат, поддерживаемый Pillow, автоматически конвертируется в RGB), выход — файл формата JSON с вероятностями предсказанных классов.

3. **Технологический стек:**
   - используются фреймворки PyTorch, Torchvision, FastAPI, Uvicorn и библиотеки Pandas, NumPy, PIL, Matplotlib, Seaborn, scikit-learn, kaggle;
   - для поднятия сервиса используются команды без использования Docker.

3. **Безопасность:**
   - в проекте не имеется секретов;
   - в репозитории нет “чувствительных” данных.
